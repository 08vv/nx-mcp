1. PROJECT OVERVIEW
================================================================================

1.1 What This MCP Server Does
-----------------------------
This is a Model Context Protocol (MCP) server named "nx-mcp" that exposes
CAD/CAM automation tools for Siemens NX as MCP-compatible JSON-RPC tools. It
allows an AI assistant (e.g. Claude, via an MCP client) to programmatically
create, modify, measure, and export 3D parts inside Siemens NX.

1.2 Purpose in the Context of Siemens NX
-----------------------------------------
Siemens NX is a professional CAD/CAM/CAE software suite. This MCP server acts
as a bridge between an AI assistant and NX, enabling natural-language-driven
3D modeling. The AI can issue commands like "create a 25mm cube" or "add a
3mm fillet" and the server translates those into real NXOpen API calls.

1.3 High-Level Architecture Summary
-------------------------------------
The system has a DUAL-MODE architecture:

  ┌──────────────┐     JSON-RPC/stdio      ┌────────────────┐
  │  MCP Client  │ ────────────────────────>│  MCP Server    │
  │  (e.g.Claude)│                          │  (server.py)   │
  └──────────────┘                          └───────┬────────┘
                                                    │
                                     ┌──────────────┴──────────────┐
                                     │                             │
                              MOCK MODE (tests)           BRIDGE MODE (production)
                              Uses mock_nxopen.py         Launches run_journal.exe
                              No real NX needed           with bridge/nx_bridge.py
                                                          as a long-running child
                                                          process inside Siemens NX
                                                          ┌───────────────────┐
                                                          │ Siemens NX        │
                                                          │ (NXOpen Python)   │
                                                          │ bridge/nx_bridge  │
                                                          │ reads JSON stdin  │
                                                          │ writes JSON stdout│
                                                          └───────────────────┘

  - MOCK MODE: When NX_MCP_USE_MOCK_NXOPEN=1, tools use a mock NXOpen module
    for testing without Siemens NX installed.

  - BRIDGE MODE (production): Tools delegate to bridge/runner.py, which spawns
    run_journal.exe to run bridge/nx_bridge.py as a persistent subprocess inside
    the NX Python environment. Communication is JSON-over-stdio between the MCP
    process and the NX bridge process.

  - SOME TOOLS (export_step, export_stl, assembly tools, some measure/utility
    tools) call NXOpen directly via NXSession without the bridge. These would
    only work if the MCP server itself runs inside the NX Python environment.

4.1 Server Initialization
--------------------------
File: src/nx_mcp/server.py

The server is a simple JSON-RPC server that reads newline-delimited JSON from
stdin and writes responses to stdout. On startup:

  1. Tool modules are imported, which triggers @mcp_tool decorators to register
     all tools in the global _registry dict.
  2. Logging is configured to stderr at DEBUG level with format:
     "[nx-mcp] LEVEL message"
  3. main() logs the tool count, then enters an infinite loop reading stdin
     line by line, parsing JSON, and dispatching to handle().

4.2 Transport Method
---------------------
PRIMARY: stdio (newline-delimited JSON-RPC)
  - Input: sys.stdin (one JSON object per line)
  - Output: sys.stdout (one JSON response per line, flushed immediately)

ALTERNATIVE: HTTP POST (via http_server.py)
  - Endpoint: POST /mcp on 127.0.0.1:8765
  - Uses FastAPI + uvicorn
  - Accepts JSON body, returns JSON response
  - Delegates to the same handle() function from server.py

4.3 Capability Registration (MCP initialize)
----------------------------------------------
When the client sends an "initialize" request, the server responds with:
  {
    "protocolVersion": "2024-11-05",
    "capabilities": {"tools": {}},
    "serverInfo": {"name": "nx-mcp", "version": "0.1.0"}
  }

Only "tools" capability is advertised. No resources or prompts are declared.

4.5 Tool Registration Mechanism
---------------------------------
File: src/nx_mcp/tools/registry.py

- A global dict _registry stores all tools.
- The @mcp_tool(name, description) decorator registers a function under the
  given name with its description.
- ToolRegistry.list_tools() generates the MCP tool list format with:
    - name, description
    - inputSchema (auto-generated from function signature via inspect)
- Schema type mapping:
    - float/int annotations -> "number"
    - bool annotations -> "boolean"
    - everything else -> "string"
- Parameters without defaults are marked as "required".

8. SIEMENS NX INTEGRATION
================================================================================

8.1 Communication Architecture
-------------------------------
The server uses TWO integration methods simultaneously:

METHOD A: NX Bridge Process (primary, production)
  File: src/nx_mcp/bridge/runner.py + src/nx_mcp/bridge/nx_bridge.py

  1. runner.py manages a persistent child process:
     - Spawns: run_journal.exe bridge/nx_bridge.py
     - run_journal.exe is Siemens NX's journal execution tool
     - The bridge script runs inside NX's Python environment with full NXOpen access
     - Communication: JSON-over-stdio (newline-delimited)
     - Sending: runner writes JSON {"tool": "...", "args": {...}} to child's stdin
     - Receiving: runner reads JSON {"ok": true/false, "message"/"error": "..."} from stdout
     - Thread-safe: Uses threading.Lock for all bridge calls
     - Auto-start: Bridge subprocess is launched on first tool call
     - Timeout: subprocess.Popen with no timeout (but run_journal CLI has 120s default)

  2. bridge/nx_bridge.py (1129 lines) runs inside Siemens NX:
     - Imports NXOpen and NXOpen.UF directly at top level
     - Gets SESSION = NXOpen.Session.GetSession()
     - Gets UF_SESSION = NXOpen.UF.UFSession.GetUFSession()
     - Maintains persistent state: last_curves, last_feature, last_body, features dict, bodies dict
     - dispatch() function maps tool names to local functions
     - main() reads JSON lines from stdin, dispatches, writes JSON results to stdout

METHOD B: Direct NXSession calls (fallback/some tools)
  Files: src/nx_mcp/nx_session.py + src/nx_mcp/nxopen_loader.py

  Some tools (export_step, export_stl, assembly tools, draw_line, draw_arc,
  set_view, take_screenshot, list_features) call NXSession directly without the
  bridge. This only works if the MCP server process itself has NXOpen available.
  nxopen_loader.py handles loading:
    - If NX_MCP_USE_MOCK_NXOPEN=1: loads mock_nxopen.py
    - Otherwise: importlib.import_module("NXOpen")

METHOD C: Journal scripts (standalone, not used by MCP server directly)
  Files: src/nx_mcp/journals/*.py + src/nx_mcp/nx_bridge.py (CLI)

  Standalone Python scripts meant to be run via run_journal.exe directly.
  Used for validation, testing, and creating specific parts independently.
  The nx_bridge.py CLI (different from the bridge module) provides:
    nx-mcp-nx validate  (runs validate_nxopen.py journal)

8.5 State Management in Bridge
-------------------------------
The bridge maintains a _state dict:
  - "last_curves": list of last created curve objects (consumed by extrude/revolve)
  - "last_feature": last created feature object
  - "last_body": last created/modified body object
  - "features": dict mapping lowercase feature name -> feature object
  - "bodies": dict mapping lowercase body name -> body object

Body/feature references like "last", "previous", "last body" are resolved via
_resolve_body() and _resolve_feature() which check state first, then iterate
through work part features/bodies.

10. ERROR HANDLING & LOGGING
================================================================================

10.1 MCP-Level Error Handling (server.py)
------------------------------------------
JSON-RPC error responses use standard codes:
  - -32700: Parse error (invalid JSON from client)
  - -32601: Method not found (unknown MCP method or unknown tool name)
  - -32602: Invalid params (TypeError when calling tool with wrong arguments)
  - -32603: Internal error (any unhandled exception during tool execution)

Format: {"jsonrpc": "2.0", "id": ..., "error": {"code": -NNNNN, "message": "..."}}

10.2 Tool-Level Error Handling
-------------------------------
Every tool function is wrapped in try/except Exception:
  - Success: Returns ToolResult(message)
  - Failure: Returns ToolError(str(exception))
  - ToolResult.__str__() -> "OK: {message}"
  - ToolError.__str__() -> "ERROR: {error}"

Input validation occurs before NX calls:
  - Size/radius/dimension must be positive (> 0)
  - Pattern count must be >= 2
  - Pattern pitch must be non-zero
  - View name must be in valid set
  - Zero-length vectors rejected in measure_angle

10.3 Bridge-Level Error Handling (bridge/nx_bridge.py)
-------------------------------------------------------
  - handle_line() catches all exceptions and returns:
    {"ok": false, "error": "...", "traceback": "..."}
  - Individual tool functions return {"ok": false, "error": "..."} for validation failures
  - SelectionError is raised for body/feature/edge resolution failures
  - NXBridgeError is raised when run_journal.exe cannot be found

10.4 Bridge Process Error Handling (bridge/runner.py)
------------------------------------------------------
  - If subprocess stdin/stdout fails: returns {"ok": false, "error": str(exc)}
  - If bridge process exits unexpectedly: returns {"ok": false, "error": stderr or "NX bridge process exited"}
  - If bridge returns invalid JSON: returns {"ok": false, "error": "Invalid bridge result JSON: ..."}
  - NXBridgeProcess.reset(): terminates/kills child process with timeout

10.5 Logging Setup
-------------------
  - logging.basicConfig(level=logging.DEBUG, stream=sys.stderr,
                         format="[nx-mcp] %(levelname)s %(message)s")
  - Logger name: "nx-mcp"
  - All log output goes to stderr (so it doesn't interfere with JSON-RPC on stdout)
  - Logs on startup: "NX MCP started — {N} tools"
  - Logs each request method: "→ {method}"
  - Mock NXOpen prints "[MOCK] ..." messages to stdout (debugging aid)

11.3 Functional Limitations (from codebase analysis + aaa.txt)
---------------------------------------------------------------
- Assembly tools are placeholder-level (don't actually position or constrain)
- export_step and export_stl don't use bridge mode (require MCP process in NX Python)
- draw_line and draw_arc don't use bridge mode
- measure tools are pure math, not NX measurements
- get_bounding_box is a placeholder (returns part reference, not coordinates)
- Sketch creation doesn't create real NX Sketch objects (just sets plane reference)
- No undo mark management around operations
- No builder cleanup guarantees on partial failure
- Body/feature resolution is name-based and can be fragile
- Complex organic/freeform shapes not supported
- draw_cylinder_profile is in bridge but not exposed as MCP tool
- No support for NX expressions/variables
- No support for NX drawing/drafting
- Edge selection heuristics (vertical/top/bottom) work by Z-coordinate analysis
  and may not work for arbitrarily oriented bodies
- fastapi and uvicorn are imported but not declared as dependencies

11.4 Test Coverage Notes
--------------------------
- 38+ test functions in test_tools.py covering all registered tools
- Tests run with mock NXOpen (no real NX needed)
- Tests verify return types (ToolResult/ToolError) and message content
- Bridge routing test verifies mock calls are forwarded correctly
- Input validation tests verify rejection of invalid parameters
